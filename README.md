# Introduction

**CanvasOAuth** is a Laravel package that provides a complete implementation of the Canvas LMS OAuth authorization flow.

The package handles the entire OAuth lifecycle for you, including:

- Authorization URL generation
- Access token storage
- Automatic access token refresh
- Token invalidation when refresh fails
- Multi-tenant support for multiple Canvas instances

This package focuses exclusively on OAuth token management. It **does not authenticate or manage your application's users**.

---

# Requirements

- PHP >= 8.0
- Laravel >= 8.0

---

# Installation

## 1. Install the package

```bash
composer require xcesaralejandro/canvasoauth
```

## 2. Publish the package resources

```bash
php artisan vendor:publish --provider="xcesaralejandro\canvasoauth\Providers\CanvasOauthServiceProvider" --force
```

## 3. Run the migrations

```bash
php artisan migrate
```

# Configuration

## Register your Canvas clients (Multi-tenant)

**CanvasOAuth 2.0** supports multiple Canvas instances simultaneously.

Each Canvas instance is stored in the database and identified by a unique **internal client code**.

When creating your **Developer Key** in Canvas, configure the following Redirect URI:

```text
https://YOUR_DOMAIN/canvas-oauth/callback
```

Then register the client by running:

```bash
php artisan canvas:create-client
```

The command will ask for the following information:

| Field | Description |
|--------|-------------|
| Internal Client Code | A unique identifier used by your application to reference this Canvas instance. |
| Canvas Base URL | The root URL of your Canvas instance (for example `https://institution.instructure.com`). |
| Client ID | The Developer Key ID generated by Canvas. |
| Client Secret | The Developer Key Secret generated by Canvas. |

After the registration is complete, the command will:

- Save the client configuration in the database.
- Display a summary of the registered client.
- Generate and display the Authorization URL for that Canvas instance.

### Why is the internal client code important?

Since the package supports multiple Canvas instances, your application must specify which Canvas client should be used whenever an OAuth flow is started.

This allows your application to support scenarios such as:

- Multiple educational institutions
- Multi-tenant SaaS applications
- Development, staging, and production Canvas environments

# Usage

## Understanding user management

CanvasOAuth manages **OAuth tokens only**.

It does **not** authenticate users or provide user management for your application.

Instead, the package stores the Canvas users that authorize your application. If your application already has its own user model, you should associate your local users with the Canvas users returned by the OAuth process.

A common approach is to add a `canvas_user_id` column to your users table and store the package-generated identifier (`$user->standard->id`).

> **Important:** Do not use the Canvas numeric user ID (`canvas_id`) as a global identifier. Different Canvas instances may contain users with identical numeric IDs. The package-generated identifier is globally unique across all registered Canvas instances.

---

## Starting the authorization flow

Retrieve the desired Canvas client and generate its authorization URL:

```php
use App\Models\CanvasClient;

$client = CanvasClient::where('code', 'YOUR_CLIENT_CODE')->firstOrFail();

$url = $client->getAuthorizationUrl();
```

Use this URL wherever it best fits your application:

- A "Connect with Canvas" button
- A hyperlink
- An automatic redirect
- Any custom authorization workflow

Remember that this process **authorizes your application to access Canvas on behalf of the user**. It is **not** intended to authenticate users into your application.

Once the user grants permission, the package stores the tokens and automatically refreshes them whenever necessary. In most cases, users only need to authorize your application once.

# Handling OAuth callbacks

When you publish the provider, a controller is included:

```php
<?php

namespace App\Http\Controllers;

use Illuminate\Http\Request;
use xcesaralejandro\canvasoauth\DataStructures\AuthenticatedUser;
use xcesaralejandro\canvasoauth\Http\Controllers\CanvasOAuthController as BaseCanvasOAuthController;

class CanvasOAuthController extends BaseCanvasOAuthController
{
    public function onPermissionGranted(AuthenticatedUser $user, Request $request): mixed
    {
        // Called after the user successfully grants access.

        return parent::onPermissionGranted($user, $request);
    }

    public function onPermissionDenied(Request $request): mixed
    {
        // Called when the user denies or cancels the authorization request.

        return parent::onPermissionDenied($request);
    }

    public function onError(\Exception $exception): mixed
    {
        // Called whenever an unexpected error occurs during the OAuth flow.

        return parent::onError($exception);
    }
}
```

The parent implementation only writes debug information to Laravel's log. If you do not need this behavior, you may safely remove the `parent::...()` calls.

# The AuthenticatedUser object

When the OAuth authorization process completes successfully, the package provides an `AuthenticatedUser` instance.

```php
class AuthenticatedUser
{
    public CanvasUser $standard;
    public ?CanvasUser $supplanted_by;
}
```

The object contains the following properties:

| Property | Description |
|----------|-------------|
| `standard` | The Canvas user who originally granted access to your application. In most cases, this is the user you should associate with your local user account. |
| `supplanted_by` | The Canvas user currently acting on behalf of the standard user. This property is `null` unless Canvas is using the **Masquerading** feature. |

For most applications, the recommended identifier is:

```php
$user->standard->id
```

This identifier is generated by the package and is guaranteed to be unique across every registered Canvas instance.

# Retrieving an access token

Access tokens are managed automatically by the package.

Whenever you request a token, the package verifies whether the current access token is still valid. If it has expired, it automatically refreshes it before returning the new value.

```php
public function onPermissionGranted(AuthenticatedUser $user, Request $request): mixed
{
    $accessToken = $user->standard->token->freshToken();

    dd($accessToken);
}
```

The `freshToken()` method always attempts to return a valid access token.

If the refresh operation fails—for example, because the refresh token has expired or the user revoked your application's authorization—the stored token is automatically removed. In that case, simply redirect the user through the OAuth authorization flow again.

# Summary

CanvasOAuth handles the complete OAuth lifecycle for Canvas LMS, including:

- Multi-tenant Canvas client management
- Authorization URL generation
- OAuth callback handling
- Secure token storage
- Automatic access token refresh
- Automatic cleanup of invalid tokens

Your application only needs to:

1. Register one or more Canvas clients.
2. Redirect users to the generated Authorization URL.
3. Associate your local users with the package-generated Canvas user identifier.
4. Call `freshToken()` whenever you need to access the Canvas API.
